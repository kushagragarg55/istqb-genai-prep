How LLMs work

Large Language Models — the engine behind all GenAI systems

An LLM is a neural network trained on massive text corpora to predict the next token in a sequence. It doesn't "know" facts — it learns statistical patterns. At inference time, it generates one token at a time, each time sampling from a probability distribution over its entire vocabulary (~50,000+ tokens).

The key testing implication: the same input can produce different outputs on different runs, because the generation process involves probabilistic sampling. This is the foundational challenge that makes GenAI testing different from classical software testing.

Prompt: "The test result was"
Context window
The maximum number of tokens the model can process at once — both input and output. Typically 4K–200K tokens. Exceeding it truncates early content.
Next-token prediction
The core LLM operation: given all previous tokens, predict the probability distribution of the next one. Responses are built by repeating this step.
Vocabulary
The set of all possible tokens (~50K). Each token gets a probability score at each generation step. The model picks one via sampling.
Pre-training vs inference
Pre-training learns patterns from data (done once, expensive). Inference generates responses (done per query, faster). Testers interact only at inference.
Exam note (K2): LLMs are stochastic by design. Classical test assertions (expected output == actual output) fail for high-temperature settings. Always specify how determinism is achieved before asserting output equality.

Temperature & sampling

The single most important parameter for testability

Temperature scales the probability distribution before sampling. At temperature=0, the model always picks the highest-probability token — fully deterministic. As temperature increases, the distribution flattens and lower-probability tokens get more chances, producing more varied and creative (but less predictable) outputs.

Tokenization

How text becomes numbers

Text is split into tokens — sub-word units — before an LLM processes it. Tokenization is not word-by-word: common words are usually one token, rare words split into multiple. Numbers, special characters, code, and non-English text often tokenize unpredictably.

Why testers care: context window limits are measured in tokens, not words. A 4,096-token context window holds roughly 3,000 English words — but only ~1,500 words if the text contains lots of code or technical terms.

04

Embeddings & vector search

Semantic similarity in high-dimensional space

An embedding converts text into a dense vector of numbers (e.g. 1,536 dimensions for OpenAI's ada-002). Semantically similar text maps to geometrically nearby points. This powers semantic search — finding relevant documents even when they use different words than the query.

Why testers care: The quality of the embedding model directly determines retrieval quality in RAG systems. A poor embedding model maps semantically equivalent queries to distant vectors — causing retrieval failure even with a perfect knowledge base.

Cosine similarity ignores vector magnitude and measures only the angle between vectors. Two texts can use different words but land at similar angles if they convey the same meaning.
05

RAG pipeline

The most-tested architecture in CT-GenAI

Retrieval-Augmented Generation (RAG) grounds LLM responses in external documents. Instead of relying purely on training data, the system retrieves relevant chunks at query time and injects them into the prompt as context. The LLM then answers based on that context — reducing hallucination risk.

The exam will describe RAG scenarios and ask you to identify which stage failed. The key is to distinguish retrieval failures from generation failures — they require completely different fixes.

RAG evaluation — the three-axis model
Every RAG response must be evaluated on all three axes independently
Context relevance
Did retriever fetch the right chunks? → Precision@k, Recall@k
Faithfulness
Is the answer grounded in context? → Faithfulness score, NLI
Answer relevance
Does it address the question? → ROUGE-L, BERTScore, human eval
Exam trap: Low faithfulness ≠ bad retrieval. If the faithfulness score is low, the LLM ignored the retrieved context. If context relevance is low, the retriever fetched wrong chunks. They are separate failure modes requiring separate fixes.

Agentic tool-use

LLMs that plan, act, and loop — a new class of test target

A GenAI agent can call external tools (APIs, databases, code runners, browsers) and chain multiple tool calls to complete a goal autonomously. This introduces testing challenges that don't exist in simple prompt-response systems: side effects, error propagation, and non-deterministic multi-step paths.

Diffusion models

Image generation — a brief but examinable topic

Diffusion models (DALL-E, Stable Diffusion, Midjourney) generate images by learning to reverse a noise-adding process. They start with pure noise and iteratively denoise it, guided by a text prompt, until an image emerges. Random noise is sampled at each run — so the same prompt produces different images every time.

Non-determinism
Each run samples fresh noise → different image. Same prompt ≠ same output. Classical pixel-exact assertions don't work.
Quality attributes
Prompt adherence, aesthetic quality, bias (representation), toxicity (NSFW content), style consistency.
Testing approach
Semantic similarity metrics, human evaluation rubrics, automated classifiers for bias/toxicity, CLIP score for prompt adherence.
Seed control
Most diffusion APIs accept a seed parameter — fixing seed makes generation deterministic. Critical for regression testing image features.
08

Model types

Foundation, fine-tuned, instruction-tuned — know the distinctions

The exam expects you to distinguish between three model types and understand the testing implications of each. Most deployed LLMs you'll encounter are instruction-tuned — but the testing strategy changes depending on what has been done to the base model.

⬡
Foundation model
Trained on massive unlabelled text corpora to predict next tokens. General purpose. Enormous parameter count. Raw capability, no task-specific alignment.
GPT-4 base, Llama 3, Gemini Ultra, Claude 3
Testing scope: capability benchmarks, knowledge cutoff verification, context window limits.
⬡⬡
Fine-tuned model
A foundation model further trained on domain-specific data to specialise its knowledge or behaviour. Risks catastrophic forgetting of base capabilities.
HR policy assistant, legal document analyser, medical diagnosis aid
Testing scope: new domain behaviour AND base capability regression. Both must pass.
⬡⬡⬡
Instruction-tuned
Fine-tuned specifically to follow human instructions using RLHF or similar. Adds safety alignment, helpfulness optimisation. Most deployed consumer LLMs are this type.
ChatGPT, Claude (Anthropic), Gemini (Google)
Testing scope: instruction following, safety guardrails, refusal behaviour, jailbreak resistance.
Fine-tuning risk: catastrophic forgetting
Performance on base capabilities vs domain-specific tasks after fine-tuning
Fine-tuning on domain data can degrade general capabilities. Regression testing after fine-tuning must cover both the new domain behaviour and pre-existing base capabilities.
